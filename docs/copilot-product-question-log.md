# Copilot 产品理解问题记录

日期：2026-04-26
用途：记录围绕“企业办公效率 Copilot”逐步形成的产品理解，后续重构代码、白皮书、Demo 路线和 Benchmark 时优先参考本文。

## 1. 背景

当前仓库已经具备本地 Memory Engine、飞书 Bot、interactive card、Bitable 同步、文档 ingestion 和 D7 抗干扰评测等能力。但这些能力如果只以 `/remember`、`/recall`、OpenClaw 插件或 CLI 工具的形式呈现，产品感会偏“手动工具”，不够像主动帮助用户的 Copilot。

新的产品理解是：

> 目标不是做一个聊天记录搜索器，也不是只做一个 OpenClaw 插件，而是做一个能提高企业办公效率的 Feishu Memory Copilot。它能省去用户翻各种群聊、文档、会议纪要去找某条消息的时间，并在合适时机主动提供帮助。

## 2. 用户提出的问题清单

### 问题 1：赛题文档和当前仓库之间，还能补充什么？

用户原始问题：

> 读一遍这个赛题文档，根据当前仓库代码的实现情况，有没有什么可以补充的内容？

当前结论：

- 赛题核心是 Define it、Build it、Prove it。
- 当前仓库最贴合方向 B“飞书项目决策与上下文记忆”和方向 D“团队知识断层与遗忘预警”。
- 需要补充的不只是功能，而是把当前能力包装成更清晰的参赛叙事：
  - 记忆定义：当前有效结论 + 状态 + 版本 + 来源证据 + 覆盖关系。
  - 已实现能力对照：CLI、本地 SQLite、飞书 Bot、interactive card、Bitable、文档 ingestion、Benchmark。
  - 证明价值：抗干扰、矛盾更新、效能指标。
  - 当前限制：白皮书未完成、D8/D9 专项评测待补、遗忘预警仍待实现。

对后续重构的影响：

- 代码重构不能只围绕命令入口，而要围绕“记忆生命周期”和“证据可解释性”整理模块。
- 白皮书应把当前实现映射到赛题三问，而不是只罗列文件和命令。

### 问题 2：作为产品，最终应该以什么形式发布？

用户原始问题：

> 作为一个产品 这个项目最终应该以什么形式发布？openclaw 插件？还是 skill？还是一个 npm 包？

当前结论：

- OpenClaw 插件、Skill、npm 包都只是入口或分发形态，不是产品本体。
- 更合理的产品本体是 Feishu Memory Engine / Feishu Memory Copilot。
- 推荐形态：
  - 主产品：飞书 Bot + interactive card + Bitable 看板。
  - 核心引擎：Memory Engine，负责抽取、存储、更新、召回、证据和评测。
  - 可复现入口：Python CLI 和 Benchmark。
  - 生态入口：OpenClaw 插件。
  - Agent 适配：Skill。
  - npm 包暂时后置，除非 OpenClaw 插件生态需要 JS/TS 包装。

对后续重构的影响：

- 核心能力应从“CLI 命令实现”中抽出来，形成稳定的应用服务层。
- OpenClaw 插件和 Skill 应作为 adapter，不应反过来决定核心数据模型。

### 问题 3：当前方案感觉不够智能，问题在哪里？

用户原始反馈：

> 这样做给我的感觉是不够智能

当前结论：

- 如果产品只是用户手动 `/remember`、手动 `/recall`，它更像知识库或记事本。
- 智能感来自主动能力：
  - 自动发现值得长期记住的信息。
  - 自动生成候选记忆卡片。
  - 自动识别冲突和覆盖关系。
  - 在讨论相关问题时主动召回历史决策。
  - 对长期未复核但重要的信息做遗忘预警。
  - 学习团队偏好和工作流规律。

对后续重构的影响：

- `/remember` 只能作为底层手动写入能力，不能作为唯一主链路。
- 需要新增“自动候选识别”和“主动召回决策”两条主线。
- 飞书卡片不只是回复样式，而是候选确认、覆盖确认、来源追溯和主动提醒的交互载体。

### 问题 4：如果要实现企业办公效率 Copilot，架构应该如何设计？

用户原始问题：

> 我需要的是一个提高企业办公效率 ，能够省去我翻各种群聊去找某一条消息的时间，同时主动提供帮助的 copilot 如果要实现这样的一个产品 架构上应该如何设计？

当前结论：

目标产品应定义为：

> 一个企业办公 Copilot，自动理解飞书里的协作上下文，把值得长期保留的信息变成团队记忆，并在用户需要前主动提醒。

推荐架构：

```text
Feishu / Docs / Meeting / Task / CLI
        |
        v
Event Ingestion Gateway
去重、权限、标准化、原始归档
        |
        v
Memory Intelligence Layer
候选识别、类型分类、重要性判断、冲突检测、摘要、证据抽取
        |
        v
Memory Governance Layer
candidate / active / superseded / rejected / stale
人工确认、版本链、复核、遗忘提醒
        |
        v
Memory Store
raw_events + memory_ledger + versions + evidence + indexes
        |
        v
Retrieval & Proactive Engine
被动问答、主动召回、会议前准备、遗忘预警
        |
        v
Product Surfaces
飞书 Bot / 卡片 / Bitable / CLI / OpenClaw Plugin / Agent Skill
```

关键设计原则：

- 原始消息不等于记忆。
- 默认召回应查 active memory，而不是直接搜索所有聊天。
- 每条回答都应带来源、版本和状态。
- 主动提醒必须有相关度阈值、冷却机制、权限检查和敏感信息脱敏。
- 记忆继承来源权限，跨群共享必须显式授权。

对后续重构的影响：

- 需要把当前代码中的 event、memory、version、evidence、publisher、benchmark 边界进一步清晰化。
- 需要新增 proactive engine，不应把主动提醒逻辑塞进飞书 handler。
- 需要把权限和安全从展示层前移到记忆写入和召回策略层。

### 问题 5：如何定义使用场景？

用户原始问题：

> 那我们现在开始聊聊怎么按照 copilot 的架构进行项目的展开，首先是如何定义使用场景

当前结论：

使用场景不应从“有哪些功能”开始，而应从企业办公中的上下文丢失问题开始：

> 哪些时刻用户因为找不到、想不起、没人提醒、旧信息混入新信息而浪费时间或做错事？

使用场景定义模板：

```text
1. 用户是谁？
2. 他正在做什么工作？
3. 缺失了什么上下文？
4. 这些上下文原本散落在哪里？
5. Copilot 应该如何介入？
6. 介入后用户节省了什么？
7. 错误介入会带来什么风险？
```

建议 P0 场景：

1. 历史决策找回
   - 用户想知道之前定过什么、为什么这么定、谁确认过。
   - Copilot 返回当前有效决策、理由、来源、版本和覆盖状态。

2. 冲突更新识别
   - 群里出现“刚才说错了”“统一改成”“旧规则不用了”等表达。
   - Copilot 识别覆盖关系，展示旧值到新值，并进入确认或自动更新流程。

3. 主动上下文提醒
   - 群里重新讨论已有历史结论的问题。
   - Copilot 主动推送当前有效记忆，减少重复争论和误用旧信息。

后续 P1/P2 场景：

- 会前或任务前准备：自动整理会议相关历史决策、待办、风险和文档。
- 遗忘预警和复核提醒：重要信息长时间未被 recall 或即将影响任务时主动提醒。
- 团队偏好学习：沉淀团队常用流程、收件人、视图偏好和安全习惯。

对后续重构的影响：

- P0 不应继续扩散到所有办公场景，应先证明“自然讨论 -> 自动候选 -> 冲突更新 -> 主动召回”的闭环。
- Benchmark 需要新增办公效率指标，不只测 Recall@K：
  - 找历史决策耗时。
  - 旧值泄漏率。
  - 版本链覆盖率。
  - 主动提醒接受率。
  - 误打扰率。
  - 证据覆盖率。

## 3. 当前产品定位草案

建议后续统一使用这个定位：

> Feishu Memory Copilot 是一个面向企业协作的主动式记忆助手。它从飞书群聊、文档、会议和任务中自动识别高价值上下文，维护带证据和版本的团队记忆，并在后续讨论、决策冲突和遗忘风险出现时主动提醒团队。

更短的答辩版本：

> 它帮团队少翻群聊、少重复争论、少用旧信息，把散落在飞书里的关键上下文变成可追溯、可更新、会主动提醒的团队记忆。

## 4. 后续重构方向

优先级从高到低：

1. 场景先行
   - 先写清 P0 三个用户旅程：历史决策找回、冲突更新识别、主动上下文提醒。
   - 每个旅程都要包含用户输入、系统判断、卡片输出、用户操作、后台状态变化和成功指标。

2. 抽出应用服务层
   - 将当前 CLI / Feishu handler 中的业务逻辑收敛到可复用 service。
   - OpenClaw plugin、Skill、CLI、Bot 都调用同一套 service。

3. 新增自动候选识别
   - 从 raw event 中判断是否值得记。
   - 输出 candidate memory，而不是直接 active。
   - 低置信度和高风险内容走人工确认。

4. 新增主动提醒策略
   - 基于当前消息、active memory、相关度、重要性、冷却时间和权限判断是否插话。
   - 主动提醒必须能解释为什么提醒。

5. 补齐评测
   - 在现有 D7 抗干扰基础上继续补 D8 矛盾更新和 D9 效能指标。
   - 为主动提醒增加误打扰率和接受率指标。

6. 白皮书重写
   - 以 Feishu Memory Copilot 为主线重写，而不是以命令集合为主线。
   - 把 OpenClaw plugin、Skill、npm wrapper 都降级为产品入口和生态适配。

## 5. 当前不做的事

- 不把 npm 包作为主发布形态。
- 不把 Skill 当作完整产品。
- 不为了显得智能而直接把所有聊天塞进 prompt 或向量库。
- 不把主动提醒做成无阈值的刷屏机器人。
- 不在 P0 阶段铺开所有飞书模块，应先证明三个核心场景闭环。
