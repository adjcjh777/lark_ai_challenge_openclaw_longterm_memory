# 飞书测试群 Live 测试剧本：OpenClaw 权限审批流功能开发

## 概述

| 项目 | 说明 |
|---|---|
| **测试目标** | 验证 Feishu Memory Copilot 在真实群聊中的记忆能力 |
| **核心攻击点** | Conflict 冲突更新 + Stale Leakage 旧值泄漏（当前最薄弱：40% / 0.429） |
| **覆盖能力** | Candidate 识别、Recall 召回、Conflict 冲突更新、版本解释、Prefetch 上下文包、敏感信息脱敏 |
| **预计耗时** | 约 15 分钟 |
| **参与角色** | PM（你，真人）、研发 A/B（模拟）、Memory Bot（被测） |

## 背景设定

你们是一个 4 人小团队，正在为 OpenClaw Agent 开发新功能：**权限审批流**。

当前问题：Agent 操作超出权限时直接 fail-closed，用户体验差。
目标：改成可审批流程，用户在飞书卡片上确认或拒绝授权。

这个功能涉及技术选型和方案变更，会产生**决策冲突更新**，正好考验 Memory Copilot 最薄弱的能力。

## 测试脚本

### Day 1：需求讨论 + 初始决策（播种期）

#### 消息 1 — PM 发起需求

**发送者**：你（PM）

```
大家好，接下来我们要做一个新功能：OpenClaw 权限审批流。
背景是现在 Agent 操作超出权限时直接 fail-closed，用户体验很差。
目标是改成可审批的流程，用户可以确认或拒绝授权。
```

#### 消息 2 — PM 细化方向

**发送者**：你（PM） | **间隔**：等 30 秒

```
具体来说，我的想法是用户在飞书里收到一张卡片，显示"Agent 请求执行 XXX 操作"，
用户点"确认"就授权，点"拒绝"就取消。
大家看看这个方向怎么样？
```

#### 消息 3 — 研发提出方案

**发送者**：你（模拟研发） | **间隔**：等 30 秒

```
【研发】PM 方向没问题。技术上我建议用飞书审批 API 做后端审批流，
前端用 interactive card 做交互。审批 API 有现成的 webhook 回调，
不用自己维护状态机。预计工期 5 天。
```

#### 消息 4 — PM 确认初始决策

**发送者**：你（PM） | **间隔**：等 30 秒

```
好，那初步定下来了：
1. 技术方案：飞书审批 API + interactive card
2. 工期：5 天
3. 后端负责审批流，前端负责卡片交互
这个决策帮我记一下。
```

⏸️ **等 1 分钟，观察 Bot 是否创建了 candidate 记忆**

> **测试点**：Bot 应创建 candidate，包含决策内容 + 证据来源（飞书消息）

### Day 2：决策变更 + 冲突更新（关键测试日）

#### 消息 5 — 研发发现问题

**发送者**：你（模拟研发） | **间隔**：等 30 秒

```
【研发】昨天说的方案我跑了一下 POC，发现问题了。
飞书审批 API 的 webhook 回调延迟平均 3 秒，峰值能到 5 秒。
用户点确认后要等好几秒才看到结果，体验太差了。
```

#### 消息 6 — 研发提出替代方案

**发送者**：你（模拟研发） | **间隔**：等 30 秒

```
【研发】我改个方案：不用审批 API 了，改成直接在飞书消息里用 interactive card 做确认。
用户点按钮后我们同步处理，响应时间可以控制在 500ms 以内。
工期也可以从 5 天缩短到 3 天，因为省去了审批 API 对接的工作量。
```

#### 消息 7 — PM 确认方案变更

**发送者**：你（PM） | **间隔**：等 30 秒

```
同意，延迟确实不能接受。那就改方案：
1. 技术方案改为：飞书消息 interactive card 直接确认（不用审批 API）
2. 工期调整为 3 天
3. 原因：审批 API webhook 延迟太高
更新一下之前的决策。
```

⏸️ **等 1 分钟，观察 Bot 如何处理冲突更新**

> **测试点**：Bot 应识别冲突，旧值（审批 API / 5 天）→ superseded，新值成为 active

## 预期收益

本场景**集中攻击 Conflict + Stale Leakage 两个最薄弱指标**（当前分别 40% 和 0.429），同时覆盖：

- **Candidate 识别**：决策类消息是否被正确识别为值得记忆
- **Recall 召回**：自然语言提问能否返回正确答案
- **Conflict 冲突更新**：方案变更时版本链是否正确维护
- **Stale Leakage**：旧值是否被正确过滤
- **Prefetch**：任务前上下文包是否结构化、完整
- **版本解释**：变更原因是否可追溯
- **安全性**：敏感信息脱敏、无关消息过滤
- **端到端链路**：消息事件 → candidate 创建 → 确认/召回 的完整流程
