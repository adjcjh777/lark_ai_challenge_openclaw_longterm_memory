## 一、问题定义：企业中“记忆”的缺失

在企业协作环境中，大量关键知识以“对话形式”存在：

* 群聊中的决策（例：部署策略、负责人）
* 文档中的约定（例：流程、规范）
* 临时沟通中的更新（例：“刚才说错了，统一改成…”）

这些信息具有三个特征：

1. **非结构化**：分散在聊天、文档、评论中
2. **易过期**：决策会被更新、推翻
3. **难复用**：依赖人工搜索或再次询问

结果是：

* 重复沟通（“部署参数是多少？”）
* 错误执行（使用旧决策）
* 时间浪费（翻聊天记录）

## 二、核心定义：什么是“企业级记忆”

### 2.1 Memory 的定义

> Memory = 可被系统提取、持续更新、可被可靠召回的团队事实单元

一个 Memory 必须满足：

| 条件     | 说明                          |
| ------ | --------------------------- |
| 可表达    | 有明确 subject（主题）             |
| 可验证    | 有来源（聊天/文档）                  |
| 可更新    | 支持版本链                       |
| 可判定有效性 | active / superseded / stale |
| 可召回    | 能被查询命中                      |

---

### 2.2 Memory 的类型（当前支持）

| 类型             | 示例             |
| -------------- | -------------- |
| decision（决策）   | “生产部署使用蓝绿发布”   |
| workflow（流程）   | “上线需要先跑测试脚本”   |
| preference（偏好） | “周报用 markdown” |
| deadline（截止）   | “周五前提交报告”      |
| risk（风险）       | “不要在周五发布”      |

---

## 三、为什么不是搜索或知识库

传统方案的问题：

| 方案    | 局限            |
| ----- | ------------- |
| 关键词搜索 | 无法理解“当前有效版本”  |
| 向量检索  | 无法处理冲突更新（A→B） |
| 知识库   | 依赖人工整理        |

Memory Engine 的关键区别：

1. **显式状态管理**

   * active / superseded / stale
2. **版本链**

   * A → B → C
3. **冲突理解**

   * “刚才说错了”触发覆盖
4. **抗干扰**

   * 在大量无关信息中仍可召回

---

## 四、系统目标（比赛三大命题）

系统必须证明：

1. 飞书协作信息 → 可复用记忆
2. 冲突更新后 → 只返回当前版本
3. 用数据证明 → 提高效率

## 五、系统架构

### 5.1 总体结构

```
飞书消息 / 文档 / CLI
        ↓
  Memory Extraction
        ↓
   Memory Engine
        ↓
  Storage（SQLite）
        ↓
 Recall / Version Trace
        ↓
 Bot / CLI / Bitable
```

---

### 5.2 核心模块

#### 1. 输入层

* 飞书 Bot（群聊）
* 文档 ingestion
* CLI 命令

---

#### 2. Memory Extraction

负责从原始文本中识别：

* subject（主题）
* type（类型）
* value（内容）
* intent（新增 / 更新）

---

#### 3. Memory Engine（核心）

功能：

* 写入（remember）
* 查询（recall）
* 更新（conflict resolution）
* 版本管理（version chain）

---

#### 4. 存储层

SQLite 表：

### Memory Ledger

| 字段            | 含义                  |
| ------------- | ------------------- |
| memory_id     | 唯一标识                |
| subject       | 主题                  |
| type          | 类型                  |
| current_value | 当前值                 |
| status        | active / superseded |
| version       | 当前版本                |
| source        | 来源                  |
| updated_at    | 更新时间                |

---

### Memory Versions

记录历史：

* version_id
* memory_id
* value
* timestamp
* reason（为什么更新）

---

### 5.3 状态机

```
candidate → active → superseded → stale → archived
```

说明：

* active：当前有效
* superseded：被新版本覆盖
* stale：长时间未使用
* archived：归档

## 六、关键机制设计

### 6.1 冲突更新（核心能力）

示例：

```
A：周报发给 Alice
B：改一下，发给 Bob
```

系统行为：

* A → superseded
* B → active
* recall 只返回 B

---

### 6.2 抗干扰

在以下场景仍能正确召回：

* 插入 500+ 无关聊天
* 多轮讨论
* 混杂噪声信息

---

### 6.3 版本可解释性

支持：

```
/versions 周报收件人
```

输出：

* v1：Alice（已覆盖）
* v2：Bob（当前）

---

### 6.4 证据链

每条记忆包含：

* 来源（消息/文档）
* 摘录（quote）
* 时间

避免“系统凭空记忆”。

## 八、安全与权限

原则：

1. 最小权限
2. 不暴露 token
3. 不返回敏感内容

## 十一、为什么适用于企业（关键论证）

区别于个人笔记：

| 维度   | 个人工具 | Memory Engine |
| ---- | ---- | ------------- |
| 数据来源 | 手动   | 自动（聊天/文档）     |
| 更新机制 | 覆盖困难 | 自动版本链         |
| 协作   | 弱    | 强             |
| 可验证性 | 低    | 高             |

---

## 十二、局限与未来方向

当前局限：

* extraction 规则仍较简单
* 未使用 embedding
* 文档解析能力有限

复赛方向：

* 卡片化 UI
* 更强 extraction
* 多源融合（邮件/任务系统）

---

## 十三、结论

本项目不是一个“搜索工具”，而是：

> 一个能理解、更新并证明“团队记忆”的系统

它解决的是：

* 信息存在 ≠ 信息可用
* 记录存在 ≠ 记忆存在

通过：

* 状态机
* 版本链
* benchmark 证明

将“对话”变成“可靠记忆”。
